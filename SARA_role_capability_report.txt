## 1. Account Creation & First-Login Flow

### Account Creation Process

**How accounts are created:**
- **MSWDO Admin**: Created via Django's `createsuperuser` command (not through in-app flow)
- **MSWDO Staff & Barangay Admin**: Created by MSWDO Admin through the in-app "Create User" flow

**View and Form Involved:**
- **View**: `accounts/views.py::create_user()` (lines 132-154)
- **URL**: `/mswdo/users/create/`
- **Form**: `accounts/forms.py::CreateUserForm` (lines 10-78)
- **Permission Check**: `if request.user.role != 'MSWDO': return HttpResponseForbidden("Access Denied")`

**Fields collected during creation:**
- username, email, first_name, last_name, middle_name, suffix
- sex, birthdate, birth_place, civil_status, citizenship
- contact_number, position
- role (MSWDO_STAFF or BARANGAY only - MSWDO not selectable)
- barangay (required for BARANGAY role, forced to None for MSWDO_STAFF)

### Temporary Password Generation

**Confirmed**: Temporary password is auto-generated
- **Function**: `accounts/views.py::generate_temp_password()` (lines 128-130)
- **Implementation**: Uses `secrets.choice()` with alphanumeric characters (12 characters)
- **Setting**: `user.set_password(temp_password)` (line 143)
- **Flag Set**: `user.must_change_password = True` (line 145)

**Display**: The temporary password is displayed ONCE to the MSWDO Admin on the success page (`accounts/create_user_success.html`) for manual communication to the user. No email invitation is sent.

### Forced Password Change Enforcement

**Middleware**: `accounts/middleware.py::ForcePasswordChangeMiddleware` (lines 4-20)

**Enforcement Logic:**
```python
if request.user.is_authenticated and getattr(request.user, 'must_change_password', False):
    allowed_paths = [reverse('force_password_change'), reverse('logout')]
    if request.path not in allowed_paths and not request.path.startswith('/static/') and not request.path.startswith('/media/'):
        return redirect('force_password_change')
```

**Security Assessment**: ✅ **Cannot bypass** - The middleware intercepts ALL requests except:
- `/force-password-change/` (the password change page itself)
- `/logout/` (allows logout)
- Static files and media

A user with `must_change_password=True` cannot navigate directly to any other URL, including dashboards or any protected endpoints.

### Password Change Flow

**View**: `accounts/views.py::force_password_change()` (lines 157-186)

**Process:**
1. User enters email + temporary password at `/login/`
2. `login_view()` checks `user.must_change_password` (line 68)
3. If True, redirects to `force_password_change` view
4. User must enter new_password + confirm_password (minimum 8 characters)
5. On success: `user.must_change_password = False`, `update_session_auth_hash()` called
6. Redirected to role-specific dashboard:
   - MSWDO → `mswdo_dashboard`
   - MSWDO_STAFF → `staff_dashboard`
   - BARANGAY → `barangay_dashboard`

### Post-Login Redirects

**From `login_view()` (lines 68-76):**
- **MSWDO**: `/mswdo/dashboard/`
- **MSWDO_STAFF**: `/staff/dashboard/`
- **BARANGAY**: `/barangay/dashboard/`

## 7. Security Observations & Potential Issues

### ✅ Properly Implemented

1. **Forced Password Change Middleware**: Correctly blocks all URLs except password change and logout. Cannot be bypassed by direct navigation.

2. **Barangay Scoping Consistency**: All Barangay Admin views properly filter by `request.user.barangay`. No cross-barangay data leakage detected.

3. **MSWDO Staff Claims Scoping**: Claims monitoring and live claims properly scoped to `created_by=request.user`.

4. **RFID Scan Scoping**: Barangay Admin RFID scan properly filters families by `household__barangay=request.user.barangay`.

5. **Decorator Usage**: `@mswdo_or_staff_required` decorator consistently applied to views intended for both MSWDO and Staff roles.

### ⚠️ Potential Concerns (Read-Only Findings)

1. **MSWDO Staff Analytics Access**: The `analytics()` view (line 349 in monitoring/views.py) uses `@mswdo_or_staff_required` but does NOT apply any scoping. MSWDO Staff can view global analytics data across all barangays and all claims (not just their own). This may be intentional but worth reviewing.

2. **MSWDO Staff Reports Access**: The `aid_reports()` view (line 38 in reports/views.py) uses `@mswdo_or_staff_required` and shows all schedules with claims. No scoping is applied to limit MSWDO Staff to their own processed claims or specific barangays.

3. **MSWDO Staff Household Map Access**: While the household map view applies role-based filtering for Barangay Admin (lines 492-494), MSWDO Staff sees ALL barangays' households (line 496). This is read-only access but provides global visibility.

4. **MSWDO Staff RFID Registration**: MSWDO Staff can register RFID cards to ANY family globally (no barangay scoping in `register_rfid()` view). This may be intentional for operational flexibility but allows cross-barangay RFID assignment.

5. **Barangay Admin Sidebar Dead Links**: Several sidebar items for Barangay Admin have no actual links (Zones, Households, Families). These appear to be placeholders and may confuse users.

6. **MSWDO Admin Sidebar Dead Links**: "Beneficiary Selection" and "Aid History" have no links in the sidebar for both MSWDO Admin and Staff.

7. **No View-Level Scoping for MSWDO Staff in Some Monitoring Views**: 
   - `aid_type_list()`: No scoping (shows all programs globally)
   - `aid_barangay_list()`: No scoping (shows all barangays)
   - `aid_barangay_detail()`: No scoping (shows all families in selected barangay)
   
   These are read-only views but provide MSWDO Staff with global visibility across all barangays.

8. **Distribution Scan Recent Claims**: While the recent claims list in `scan_rfid()` is scoped for MSWDO Staff (line 311-312), the view does not prevent MSWDO Staff from filtering by ANY barangay via GET parameter (line 308-309). The scoping only applies to the claims queryset, not the barangay filter dropdown.

### 🔒 Security Strengths

1. **Middleware Enforcement**: Password change cannot be bypassed - properly implemented at middleware level.

2. **Consistent Role Checks**: All views use explicit `if request.user.role != 'X': return HttpResponseForbidden()` patterns.

3. **Queryset Scoping**: Barangay Admin views consistently filter querysets by `request.user.barangay`.

4. **No Direct Object Reference Attacks**: All views use `get_object_or_404` with scoping parameters, preventing ID enumeration attacks.

5. **Session Protection**: `@session_protected` decorator applied to sensitive views ensures authentication.

## Summary

The S.A.R.A. system implements a three-tier role-based access control model with clear separation of concerns:

- **MSWDO Admin**: Full system access with global visibility across all barangays and all operational capabilities.
- **MSWDO Staff**: Operational access for distribution and household viewing (read-only globally), with claims monitoring scoped to their own processed claims. Cannot manage users or configure programs.
- **Barangay Admin**: Full CRUD access to household/family/member data within their assigned barangay only. Can scan RFID for claims within their barangay. Cannot access system configuration or other barangays' data.

The forced password change mechanism is properly implemented and cannot be bypassed. Barangay scoping is consistently applied for Barangay Admin views. MSWDO Staff has appropriate scoping for claims processing but retains global visibility for analytics and reports, which may be intentional for oversight purposes.
